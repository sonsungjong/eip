# [5과목] 인터페이스 구현

---

## 1. 인터페이스 설계 및 데이터 표준
송신 시스템과 수신 시스템 간의 데이터 교환을 위해 데이터 포맷, 통신 규칙, 보안 등을 정의하는 과정입니다.

* **인터페이스 명세서:** 시스템 간 인터페이스 방식, 송수신 데이터 명세, 오류 처리 방안 등을 상세히 작성한 문서.

### 데이터 포맷
| 포맷 | 설명 |
| :--- | :--- |
| **JSON** | 속성-값 쌍(Key-Value) 형태, 가볍고 파싱이 빠름. 웹/모바일, AJAX에서 주로 사용 |
| **XML** | 태그(Tag)를 사용하여 데이터를 구조화. 다소 무겁지만 확장성과 호환성이 좋음 |
| **YAML** | 사람이 읽기 쉬운 데이터 직렬화 양식. 들여쓰기로 데이터의 구조를 구분 |
| **CSV** | 쉼표(,)로 항목을 구분하는 텍스트 기반 데이터 형식 |

## 2. 인터페이스 구현 방법

### 데이터 통신을 이용한 인터페이스 구현
인터페이스 송수신 시스템 사이에서 데이터 통신을 통해 인터페이스를 구현하는 방법입니다.
* **방식:** AJAX, JSON, XML 등의 데이터 포맷을 활용하여 송수신 처리

### 인터페이스 엔티티를 이용한 인터페이스 구현
인터페이스가 필요한 시스템 사이에 별도의 인터페이스 엔티티(테이블)를 두어 데이터를 교환하는 방법입니다.
* **방식:** 송신 시스템의 인터페이스 테이블 → 수신 시스템의 인터페이스 테이블로 데이터 전송

## 2-1. EAI (Enterprise Application Integration)
기업 내 각종 애플리케이션 및 플랫폼 간의 정보를 전달, 연계, 통합하는 솔루션입니다.

### EAI 구축 유형
| 유형 | 설명 |
| :--- | :--- |
| **Point-to-Point** | 가장 기본적인 방식. 시스템 간 1:1로 직접 연결. 변경·재사용이 어려움 |
| **Hub & Spoke** | 단일 접점인 **허브 시스템**을 통해 데이터를 전송. 확장 및 유지보수 용이하지만 허브 장애 시 전체 영향 |
| **Message Bus (ESB)** | 미들웨어(버스)를 두어 처리하는 방식. 확장성이 뛰어나며 대용량 처리에 유리 |
| **Hybrid** | Hub & Spoke + Message Bus를 혼합한 방식. 유연성이 높지만 구축 비용이 큼 |

> 암기: **포·허·메·하** (Point-to-Point → Hub & Spoke → Message Bus → Hybrid)

### ESB (Enterprise Service Bus)
미들웨어를 중심으로 각 시스템을 버스 형태로 연결하여 느슨한 결합(Loose Coupling)으로 통합하는 방식입니다.
* 표준 기반 인터페이스를 통해 서비스 간 연동을 지원
* EAI와 비교: EAI는 애플리케이션 간 직접 통합, ESB는 서비스 중심의 느슨한 통합

## 2-2. 미들웨어 (Middleware)
클라이언트와 서버 간, 또는 운영체제와 응용 프로그램 사이에서 다양한 서비스를 제공하는 소프트웨어입니다.

### 미들웨어 유형
| 유형 | 설명 |
| :--- | :--- |
| **DB (Database)** | 클라이언트에서 원격 데이터베이스에 접속할 수 있게 해주는 미들웨어 |
| **RPC (Remote Procedure Call)** | 원격 프로시저를 로컬 프로시저처럼 호출할 수 있게 해주는 미들웨어 |
| **MOM (Message Oriented Middleware)** | **메시지 기반** 비동기형 미들웨어. 메시지 큐를 이용하여 메시지를 전달 |
| **TP-Monitor (Transaction Processing Monitor)** | 항공기 예약, 은행 업무 등 **온라인 트랜잭션**을 처리·감시하는 미들웨어 |
| **ORB (Object Request Broker)** | **객체 지향** 미들웨어. 코바(CORBA) 표준 스펙을 구현한 미들웨어 |
| **WAS (Web Application Server)** | **웹 환경**을 구현하기 위한 미들웨어. 동적 서버 콘텐츠 수행. 가장 많이 사용됨 (Tomcat, WebLogic 등) |

## 2-3. REST / SOAP

### SOAP (Simple Object Access Protocol)
* HTTP, HTTPS, SMTP 등을 통해 **XML 기반**의 메시지를 교환하는 프로토콜
* WSDL(웹 서비스 기술 언어)과 UDDI(레지스트리)를 사용
* 보안, 트랜잭션, 신뢰성 등 표준이 잘 정의되어 있음 (WS-Security 등)
* 상대적으로 무겁고 복잡함

### REST (Representational State Transfer)
* HTTP 프로토콜의 메서드(GET, POST, PUT, DELETE)를 그대로 활용하는 아키텍처 스타일
* **자원(Resource)** 을 URI로 표현하고, HTTP 메서드로 CRUD를 수행
* JSON, XML 등 다양한 데이터 포맷 지원, 가볍고 단순
* **RESTful API:** REST 원리를 따르는 시스템의 웹 API

| 비교 항목 | SOAP | REST |
| :--- | :--- | :--- |
| 메시지 포맷 | XML만 사용 | JSON, XML 등 다양 |
| 통신 방식 | 무겁고 복잡 | 가볍고 단순 |
| 표준 | 엄격한 표준 (WSDL, UDDI) | 느슨한 가이드라인 |
| 보안 | WS-Security 등 자체 표준 | HTTPS 기반 |

## 3. 인터페이스 데이터 보안
인터페이스 과정에서 데이터 탈취나 위변조를 막기 위해 네트워크 구간 및 데이터 자체에 보안을 적용합니다.

### 1) 네트워크 구간 보안 (전송 구간 암호화)
| 프로토콜 | 설명 |
| :--- | :--- |
| **IPSec** | 네트워크 계층(3계층)에서 IP 패킷 단위로 인증, 암호화, 키 관리를 수행하는 보안 프로토콜 |
| **SSL/TLS** | 전송 계층(4계층)과 응용 계층(7계층) 사이에서 클라이언트-서버 간 안전한 통신을 지원하는 프로토콜 |
| **S-HTTP** | 웹 상에서 HTTP 메시지 자체를 암호화하는 프로토콜 |

### 2) 데이터 암호화 알고리즘

#### 양방향 암호화 (복호화 가능)

**대칭키 (비밀키/개인키) 암호화**
암호화와 복호화에 **같은 키** 사용. 속도가 빠르지만 키 관리(전달)가 어려움.

| 알고리즘 | 블록 크기 | 키 길이 | 특징 |
| :--- | :--- | :--- | :--- |
| **DES** | 64비트 | 56비트 | 미국 구 표준, 현재 취약 |
| **3DES** | 64비트 | 112/168비트 | DES를 3번 적용 |
| **AES** | 128비트 | 128/192/256비트 | 미국 현행 표준 (DES 대체) |
| **SEED** | 128비트 | 128비트 | 국산 암호 알고리즘 |
| **ARIA** | 128비트 | 128/192/256비트 | 국산 암호 알고리즘 (경량 환경) |
| **IDEA** | 64비트 | 128비트 | 스위스에서 개발, PGP에 사용 |

* **블록 암호 방식:** 고정 크기 블록 단위로 암호화 (DES, AES, SEED 등)
* **스트림 암호 방식:** 비트/바이트 단위로 실시간 암호화 (RC4 등)

**비대칭키 (공개키) 암호화**
암호화(공개키)와 복호화(개인키)에 **다른 키** 사용. 키 관리가 쉽지만 속도가 느림.

| 알고리즘 | 기반 원리 | 특징 |
| :--- | :--- | :--- |
| **RSA** | 큰 수의 **소인수분해** 어려움 | 가장 대표적인 공개키 암호 알고리즘 |
| **ECC** | **타원 곡선** 이산 대수 문제 | RSA보다 짧은 키로 동일 보안 수준. 모바일 환경에 적합 |
| **디피-헬만(Diffie-Hellman)** | **이산 대수** 문제 | 최초의 공개키 알고리즘, 키 교환 프로토콜 |
| **ElGamal** | **이산 대수** 문제 | 디피-헬만을 발전시킨 공개키 암호 방식 |

#### 단방향 암호화 (해시, 복호화 불가능)
임의의 길이의 입력 데이터를 **고정된 길이**의 해시값으로 변환. 패스워드 저장, 무결성 검증 등에 사용.

| 알고리즘 | 해시값 길이 | 특징 |
| :--- | :--- | :--- |
| **MD5** | 128비트 | 현재 취약점 발견되어 보안 용도 권장하지 않음 |
| **SHA-1** | 160비트 | 미국 표준이었으나 충돌 취약점 발견 |
| **SHA-256** | 256비트 | 현재 가장 널리 사용되는 해시 알고리즘 |
| **HAS-160** | 160비트 | 국산 해시 알고리즘 |

## 4. 인터페이스 보안 취약점 및 대응

### 주요 보안 공격 유형
| 공격 유형 | 설명 | 대응 방안 |
| :--- | :--- | :--- |
| **SQL Injection** | 입력란에 **악의적인 SQL 구문**을 삽입하여 DB를 조작하는 공격 | 입력값 검증, PreparedStatement 사용, 특수문자 필터링 |
| **XSS (Cross-Site Scripting)** | 웹 페이지에 **악성 스크립트**를 삽입하여 사용자 정보를 탈취 | 입력값 검증, HTML 인코딩, CSP 적용 |
| **CSRF (Cross-Site Request Forgery)** | 사용자가 자신의 의지와 무관하게 공격자가 의도한 **위조 요청**을 보내게 하는 공격 | CSRF 토큰 사용, Referer 검증 |

### 시큐어 코딩 (Secure Coding)
소프트웨어 개발 과정에서 개발자의 실수, 논리적 오류 등으로 인해 소프트웨어에 내포될 수 있는 보안 취약점을 배제하기 위한 코딩 기법입니다.

**시큐어 코딩 가이드 주요 항목:**

| 항목 | 설명 |
| :--- | :--- |
| **입력 데이터 검증 및 표현** | SQL Injection, XSS 등 입력값 관련 보안 약점 방지 |
| **보안 기능** | 인증, 접근 제어, 암호화 등의 부적절한 구현 방지 |
| **시간 및 상태** | 경쟁 조건(Race Condition), 교착상태(Deadlock) 관련 보안 약점 방지 |
| **에러 처리** | 에러 메시지를 통한 정보 노출 방지 |
| **코드 오류** | Null Pointer 역참조, 부적절한 자원 해제 등 방지 |
| **캡슐화** | 잘못된 세션을 통한 데이터 노출 방지 |
| **API 오용** | 보안에 취약한 API 사용 방지 |

> 암기: **입보시에코캡API**

## 5. 인터페이스 구현 검증 (테스트 자동화 도구) 
인터페이스가 정상적으로 동작하는지 확인하기 위해 사용하는 테스트 자동화 프레임워크 및 도구들입니다.

| 도구 | 특징 및 설명 |
| :--- | :--- |
| **xUnit** | Java(JUnit), C++(CppUnit), .NET(NUnit) 등 **다양한 언어를 지원하는 단위 테스트** 프레임워크 |
| **STAF** | 서비스 호출, 컴포넌트 재사용 등 **다양한 환경(분산 환경)을 지원**하는 크로스 플랫폼 테스트 프레임워크 |
| **FitNesse** | **웹 기반** 테스트 케이스 설계, 실행, 결과 확인 등을 지원하는 테스트 프레임워크 |
| **NTAF** | FitNesse의 장점(협업)과 STAF의 장점(재사용 및 확장성)을 통합한 **NHN(네이버)**의 테스트 자동화 프레임워크 |
| **Selenium** | **웹 애플리케이션 테스트**를 위한 프레임워크. 다양한 브라우저 및 언어 지원 |
| **watir** | **Ruby(루비)** 언어 기반의 웹 애플리케이션 UI 테스트 자동화 도구 |

## 6. 인터페이스 오류 처리
인터페이스 동작 시 발생할 수 있는 오류를 식별하고 처리하는 방법입니다.

### 오류 유형
| 유형 | 설명 |
| :--- | :--- |
| **송신 오류** | 송신 시스템에서 데이터 생성 및 전송 시 발생하는 오류 |
| **수신 오류** | 수신 시스템에서 데이터 수신 및 처리 시 발생하는 오류 |
| **연계 오류** | 연계 서버(미들웨어) 또는 네트워크 구간에서 발생하는 오류 |

### 오류 처리 방법
* **오류 로그 기록:** 오류 발생 시 오류 코드, 오류 메시지, 발생 시간 등을 로그에 기록
* **오류 알림:** 관련 담당자에게 오류 발생 사실을 실시간 통보 (이메일, SMS 등)
* **오류 재처리:** 일시적 오류 시 자동 재전송(재시도) 수행

## 7. 인터페이스 감시 (Monitoring)
인터페이스의 동작 상태를 실시간으로 확인하고 관리하는 활동입니다.

### 감시 방법
| 방법 | 설명 |
| :--- | :--- |
| **트랜잭션 추적** | 인터페이스 트랜잭션의 처리 과정을 추적하여 정상 동작 여부 확인 |
| **로그 분석** | 인터페이스 동작 시 생성되는 로그를 분석하여 이상 징후 감지 |
| **임계치 설정** | 응답 시간, 처리량 등에 임계치를 설정하여 초과 시 알림 발생 |

### APM (Application Performance Management)
인터페이스 및 애플리케이션의 성능을 모니터링하고 관리하는 도구입니다.
* 데이터베이스 접속, 자원 사용량, 응답 시간 등을 시각적으로 확인하여 성능 장애 요인을 분석합니다.
* 종류: **스카우터(Scouter)**, **제니퍼(Jennifer)** 등

## ★ 핵심 암기 포인트 ★

1. **대칭키 알고리즘:** DES(56비트 키), AES(128비트 블록, 미국 표준), SEED·ARIA(국산)
2. **비대칭키 알고리즘:** RSA(소인수분해), ECC(타원곡선), 디피-헬만(이산대수, 키 교환)
3. **해시 알고리즘:** MD5(128비트), SHA-256(256비트), HAS-160(국산)
4. **대칭키 vs 비대칭키:** 대칭키=같은 키/빠름/키 전달 어려움, 비대칭키=다른 키/느림/키 관리 쉬움
5. **보안 공격 3대장:** SQL Injection, XSS, CSRF
