# [12과목] 제품 소프트웨어 패키징

---

## 1. 제품 소프트웨어 패키징
개발이 완료된 소프트웨어를 고객에게 전달하기 위해 패키징하는 과정입니다.
* **사용자 중심**으로 작성하며, 다양한 환경(OS, 하드웨어 등)에서 구동될 수 있도록 합니다.

### 릴리즈 노트 (Release Note)
소프트웨어 배포 시 추가, 변경, 버그 수정 사항 등을 상세히 담아 고객에게 제공하는 문서입니다.
* **구성 항목:** 머릿말, 개요, 목적, 문제 요약, 재현 항목, 수정/개선 내용, 사용자 영향도, SW 지원 영향도, 면책 조항, 연락처

## 2. 소프트웨어 매뉴얼

| 종류 | 설명 |
| :--- | :--- |
| **설치 매뉴얼** | 소프트웨어를 설치하기 위한 **설치 절차, 설치 환경, 패키지 내용** 등을 기술한 문서 |
| **사용자 매뉴얼** | 소프트웨어를 사용하는 데 필요한 **기능, 사용법, 주의사항** 등을 기술한 문서 |

## 3. 디지털 저작권 관리 (DRM)
디지털 콘텐츠의 불법 복제를 막고, 합법적인 사용자만 사용하도록 제어하는 기술입니다.

### DRM 구성 요소

| 구성 요소 | 설명 |
| :--- | :--- |
| **콘텐츠 제공자 (Content Provider)** | 콘텐츠를 제작하고 권리를 부여하는 주체 |
| **콘텐츠 분배자 (Distributor)** | 암호화된 콘텐츠를 유통하는 주체 |
| **클리어링 하우스 (Clearing House)** | 저작권 권한 정책, **라이선스 발급**, 결제/정산 관리를 수행 |
| **DRM 컨트롤러** | 배포된 콘텐츠의 이용 권한을 통제하는 프로그램 |
| **보안 컨테이너** | 콘텐츠 원본을 안전하게 유통하기 위한 전자적 보안 장치 |

### 패키징 도구 요소 기술

| 기술 | 설명 |
| :--- | :--- |
| **암호화 (Encryption)** | 공개키/비밀키 방식을 사용하여 콘텐츠 보호 |
| **식별 기술 (Identification)** | 콘텐츠에 식별 체계 부여 (DOI, URI 등) |
| **크랙 방지 (Anti-Cracking)** | 리버스 엔지니어링 방지. 코드 난독화, Secure DB 적용 |
| **정책 관리 (Policy Management)** | 라이선스 발급 및 사용 정책 표현 (XrML, MPEG-21 등) |
| **저작권 표현 (Rights Expression)** | 라이선스의 내용을 표현하는 언어 (XrML, ODRL) |
| **워터마킹 (Watermarking)** | 콘텐츠에 저작권 정보를 삽입하여 불법 복제 추적 |
| **핑거프린팅 (Fingerprinting)** | 콘텐츠에 구매자 정보를 삽입하여 불법 배포자를 추적 |

> **워터마킹 vs 핑거프린팅:** 워터마킹=저작권자 정보 삽입, 핑거프린팅=구매자 정보 삽입

## 4. 오픈소스 라이선스

| 라이선스 | 특징 |
| :--- | :--- |
| **GPL (GNU General Public License)** | 가장 강한 제약. 소스 코드 공개 의무. **파생 저작물도 반드시 GPL** 적용 (카피레프트) |
| **LGPL** | GPL보다 완화. 라이브러리 **링크(사용)**는 소스 공개 의무 없음 |
| **Apache License** | 소스 코드 수정 후 **수정 사항 안내 의무**, 소스 공개 의무 없음 |
| **MIT License** | 가장 자유로운 라이선스 중 하나. 저작권 표시만 하면 자유롭게 사용 |
| **BSD License** | MIT와 유사. 저작권 표시 필수. 재배포 시 소스 공개 의무 없음 |

> **카피레프트(Copyleft):** 저작물의 자유로운 사용을 보장하되, 파생 저작물에도 같은 조건을 적용하도록 요구하는 개념 (GPL의 핵심)

## 5. 소프트웨어 형상 관리 (SCM)
소프트웨어 개발 과정에서 발생하는 모든 변경 사항을 추적하고 통제하는 체계입니다.

### 형상 관리 절차 (식·통·상·감)

| 절차 | 설명 |
| :--- | :--- |
| **형상 식별** | 관리 대상(베이스라인)을 식별하고 **고유한 이름과 번호** 부여 |
| **형상 통제** | 변경 요청에 대한 검토 및 승인. **형상 통제 위원회(CCB)** 승인 필요 |
| **형상 상태 보고** | 변경 상태의 진행 정도를 기록하고 관련자에게 보고 |
| **형상 감사** | 기준선(Baseline)에 맞게 요구사항이 잘 반영되었는지 **객관적으로 검증** |

### 버전 관리 주요 용어

| 용어 | 설명 |
| :--- | :--- |
| **저장소 (Repository)** | 소프트웨어 형상 항목이 저장되는 공간 |
| **가져오기 (Import)** | 빈 저장소에 최초로 파일을 복사하는 작업 |
| **체크아웃 (Checkout)** | 저장소에서 로컬로 파일을 받아오는 작업 |
| **체크인 (Checkin)** | 변경한 파일을 저장소에 반영하는 작업 |
| **커밋 (Commit)** | 체크인 시 이전과의 변경 이력을 기록하면서 저장소에 반영 |
| **업데이트 (Update)** | 저장소의 최신 내용을 로컬로 갱신 |
| **브랜치 (Branch)** | 주 개발 라인(Trunk)에서 분기하여 **독립적인 개발 라인** 생성 |
| **머지 (Merge)** | 브랜치의 변경 사항을 **주 개발 라인에 통합** |
| **태그 (Tag)** | 특정 시점의 버전에 **이름표**를 붙여 관리 (릴리즈 버전 등) |

### 버전 관리 도구 분류

| 방식 | 설명 | 도구 |
| :--- | :--- | :--- |
| **공유 폴더 방식** | 로컬 공유 폴더에 버전 관리 | SCCS, RCS |
| **클라이언트/서버 방식 (중앙 집중형)** | 중앙 서버에 마스터 저장소. 서버 다운 시 작업 어려움 | **CVS**, **SVN** |
| **분산 저장소 방식 (분산형)** | 로컬+원격 저장소 모두 보유. 서버 문제 시에도 로컬에서 작업 가능 | **Git** |

### Git 주요 명령어 

| 명령어 | 설명 |
| :--- | :--- |
| **git clone** | 원격 저장소를 로컬로 복제 |
| **git add** | 변경 파일을 스테이징 영역에 추가 |
| **git commit** | 스테이징 영역의 변경 사항을 로컬 저장소에 기록 |
| **git push** | 로컬 저장소의 변경 사항을 **원격 저장소에 반영** |
| **git pull** | 원격 저장소의 변경 사항을 로컬로 가져와 **자동 병합** |
| **git fetch** | 원격 저장소의 변경 사항을 가져오되 **병합하지 않음** |
| **git branch** | 브랜치 생성/조회 |
| **git merge** | 브랜치를 현재 브랜치에 병합 |
| **git fork** | 다른 사람의 원격 저장소를 자신의 원격 저장소로 복제 (GitHub 기능) |

### SVN 주요 명령어

| 명령어 | 설명 |
| :--- | :--- |
| **svn checkout** | 저장소에서 로컬로 소스 받기 |
| **svn commit** | 변경 사항을 저장소에 반영 |
| **svn update** | 저장소의 최신 내용으로 갱신 |
| **svn add** | 새 파일을 버전 관리에 추가 |
| **svn diff** | 변경 사항 비교 |

## 6. 빌드 자동화 도구
소스 코드의 컴파일, 테스트, 패키징, 배포 등을 자동화하는 도구입니다.

| 도구 | 설명 |
| :--- | :--- |
| **Ant** | Java 기반. **XML** 기반 빌드 스크립트 (build.xml). 규칙이 없어 유연하지만 복잡해질 수 있음 |
| **Maven** | Java 기반. **XML** 기반 (pom.xml). 규칙과 라이프사이클 기반으로 표준화된 빌드 |
| **Gradle** | Java 기반. **Groovy/Kotlin DSL** 기반 (build.gradle). Ant의 유연함 + Maven의 규칙성 결합. **Android 공식 빌드 도구** |
| **Jenkins** | **CI/CD** 도구. 빌드, 테스트, 배포를 자동화하는 서버. 다양한 플러그인 지원 |

### CI/CD 개념

| 용어 | 설명 |
| :--- | :--- |
| **CI (Continuous Integration)** | 개발자가 변경한 코드를 **지속적으로 통합하고 빌드·테스트**하는 개발 방식 |
| **CD (Continuous Delivery/Deployment)** | CI의 확장으로, 빌드·테스트 후 **자동으로 배포**까지 수행하는 방식 |

## ★ 핵심 암기 포인트 ★

1. **형상 관리 절차:** 식별 → 통제(CCB 승인) → 상태 보고 → 감사
2. **버전 관리 방식:** 공유 폴더(RCS) → 클라이언트/서버(CVS, SVN) → 분산형(Git)
3. **Git 핵심 명령어:** clone(복제), add(스테이징), commit(로컬 기록), push(원격 반영), pull(원격→로컬+병합), fetch(원격→로컬, 병합X)
4. **DRM 클리어링 하우스:** 라이선스 발급, 결제/정산 관리
5. **워터마킹=저작권자 정보, 핑거프린팅=구매자 정보**
6. **오픈소스 라이선스:** GPL(소스 공개 의무, 카피레프트) > LGPL > Apache > MIT/BSD(자유로움)
7. **빌드 도구:** Ant(XML/유연), Maven(XML/표준), Gradle(Groovy/Android), Jenkins(CI/CD)
8. **릴리즈 노트:** 소프트웨어 배포 시 변경·수정 사항을 담은 고객 제공 문서
