# [4과목] 서버 프로그램 구현

---

## 1. 모듈 (Module)과 성능 평가
소프트웨어 설계에서 기능 단위로 분해하고 추상화되어 재사용 및 공유 가능한 수준으로 만들어진 단위입니다. 모듈의 독립성은 **결합도(Coupling)**와 **응집도(Cohesion)**에 의해 측정됩니다.

* **좋은 설계의 조건:** 결합도는 약하게(낮게), 응집도는 강하게(높게), 모듈 크기는 작게!

### 결합도 (Coupling) - 낮을수록 좋음
모듈 간에 상호 의존하는 정도입니다.
*(암기: 데·스·제·외·공·내 — 우측으로 갈수록 결합도 높아짐/나쁨)*

| 결합도 | 설명 |
| :--- | :--- |
| **데이터(Data)** | 모듈 간에 매개변수로 데이터만 주고받는 상태 (가장 좋음) |
| **스탬프(Stamp)** | 배열이나 레코드 등의 자료 구조가 전달되는 상태 |
| **제어(Control)** | 제어 신호(Control Signal)를 전달하여 통제하는 상태 |
| **외부(External)** | 어떤 모듈에서 선언한 데이터를 외부의 다른 모듈에서 참조하는 상태 |
| **공통(Common)** | 전역 변수를 참조하고 갱신하는 상태 |
| **내용(Content)** | 다른 모듈 내부에 있는 변수나 기능을 직접 참조/수정하는 상태 (가장 나쁨) |

### 응집도 (Cohesion) - 높을수록 좋음
모듈 내부의 구성 요소들이 서로 관련되어 있는 정도입니다.
*(암기: 우·논·시·절·통·순·기 — 우측으로 갈수록 응집도 높아짐/좋음)*

| 응집도 | 설명 |
| :--- | :--- |
| **우연적(Coincidental)** | 아무 관련 없는 요소들이 모인 상태 (가장 나쁨) |
| **논리적(Logical)** | 유사한 성격을 갖거나 특정 형태로 분류되는 처리 요소들이 모인 상태 |
| **시간적(Temporal)** | 특정 시간에 처리되어야 하는 활동들을 모은 상태 |
| **절차적(Procedural)** | 모듈 안의 다수 관련 기능들이 순차적으로 수행되는 상태 |
| **통신적(Communicational)** | 동일한 입출력을 사용하여 서로 다른 기능을 수행하는 상태 |
| **순차적(Sequential)** | 한 활동의 출력값을 다음 활동의 입력 데이터로 사용하는 상태 |
| **기능적(Functional)** | 모듈 내부의 모든 기능이 단일한 목적을 위해 수행되는 상태 (가장 좋음) |

### 팬인(Fan-In) / 팬아웃(Fan-Out) ★빈출★
| 구분 | 설명 | 좋은 방향 |
| :--- | :--- | :--- |
| **Fan-In** | 어떤 모듈을 **호출하는** 모듈의 수 (자신을 사용하는 상위 모듈 수) | 높을수록 좋음 (재사용성↑) |
| **Fan-Out** | 어떤 모듈이 **호출하는** 모듈의 수 (자신이 사용하는 하위 모듈 수) | 낮을수록 좋음 (복잡도↓) |

> 시스템의 복잡도를 최적화하려면: **Fan-In은 높게, Fan-Out은 낮게** 설계한다.

## 2. 공통 모듈
여러 프로그램에서 공통으로 사용할 수 있는 모듈입니다.

### 공통 모듈 명세 기법의 원칙 ★서술형 대비★
| 원칙 | 설명 |
| :--- | :--- |
| **정확성(Correctness)** | 시스템 구현 시 해당 기능이 필요하다는 것을 알 수 있도록 정확히 작성 |
| **명확성(Clarity)** | 해당 기능을 이해할 때 중의적으로 해석되지 않도록 명확히 작성 |
| **완전성(Completeness)** | 시스템 구현 시 필요한 모든 것을 기술 |
| **일관성(Consistency)** | 공통 기능들 간에 상호 충돌이 발생하지 않도록 작성 |
| **추적성(Traceability)** | 기능에 대한 요구사항의 출처, 관련 시스템 등의 관계를 파악할 수 있도록 작성 |

## 3. 코드 (Code) ★빈출★
자료의 분류, 조합, 집계, 추출을 용이하게 하기 위해 사용하는 기호입니다.

| 코드 종류 | 설명 | 예시 |
| :--- | :--- | :--- |
| **순차 코드 (Sequence Code)** | 일정 기준에 따라 차례로 일련번호를 부여 | 1, 2, 3, 4… |
| **블록 코드 (Block Code)** | 공통 성질에 따라 블록으로 구분하고, 블록 내에서 일련번호 부여 | 1001~1100: 영업부 |
| **10진 코드 (Decimal Code)** | 10진 분할 방식을 이용하여 항목에 코드 부여 (도서 분류식) | 000: 총류, 100: 철학 |
| **그룹 분류 코드 (Group Classification)** | 대·중·소 분류 등으로 구분하여 코드 부여 | 1-01-001 |
| **연상 코드 (Mnemonic Code)** | 대상 항목의 명칭이나 약호와 관계있는 숫자/문자를 코드로 부여 | TV-40: 40인치 TV |
| **표의 숫자 코드 (Significant Digit)** | 대상 항목의 성질(물리적 수치)을 코드에 그대로 반영 | 120-720-300: 사이즈 |
| **합성 코드 (Combined Code)** | 2개 이상의 코드를 조합하여 사용 | 연상코드 + 순차코드 |

## 4. 객체지향 (Object-Oriented)
현실 세계의 개체(Entity)를 속성과 메서드가 결합된 형태의 객체(Object)로 표현하는 기법입니다.

### 객체지향의 기본 구성요소

| 구성요소 | 설명 |
| :--- | :--- |
| **클래스(Class)** | 같은 종류의 집단에 속하는 속성과 행위를 정의한 것. 객체의 설계도(틀) |
| **객체(Object)** | 클래스의 인스턴스. 속성(Attribute)과 메서드(Method)로 구성 |
| **메서드(Method)** | 객체가 수행할 수 있는 동작(행위). 객체의 속성을 조작하는 연산 |
| **메시지(Message)** | 객체에게 어떤 행위를 하도록 지시하는 명령. 객체 간 상호작용 수단 |
| **인스턴스(Instance)** | 클래스에 속한 각각의 개별 객체 |
| **속성(Attribute/Property)** | 객체가 가지고 있는 데이터(상태 정보) |

### 객체지향의 특징 (캡·상·다·추·정)
* **캡슐화 (Encapsulation):** 속성과 관련된 연산을 하나로 묶고, 외부의 불법적인 접근을 막는 것. 결합도↓, 재사용 용이, 인터페이스 단순화.
* **상속 (Inheritance):** 상위 클래스의 속성과 메서드를 하위 클래스가 물려받는 것. 재사용성↑.
* **다형성 (Polymorphism):** 하나의 메시지에 대해 각 객체가 고유한 방식으로 응답할 수 있는 능력. (오버로딩, 오버라이딩)
* **추상화 (Abstraction):** 불필요한 세부사항을 제거하고 공통적인 특성을 추출하여 정의하는 것.
* **정보 은닉 (Information Hiding):** 다른 객체에게 자신의 정보를 숨기고, 자신의 연산만을 통하여 접근을 허용하는 것. 캡슐화에서 가장 중요한 개념.

### 오버로딩 vs 오버라이딩 ★매우 빈출★
| 구분 | 오버로딩 (Overloading) | 오버라이딩 (Overriding) |
| :--- | :--- | :--- |
| **정의** | 같은 이름의 메서드를 **매개변수의 유형·개수를 다르게** 하여 여러 개 정의 | 상위 클래스의 메서드를 하위 클래스에서 **재정의** |
| **메서드 이름** | 동일 | 동일 |
| **매개변수** | 다름 (개수, 타입, 순서) | 동일 |
| **리턴 타입** | 관계없음 | 동일 |
| **적용 범위** | 같은 클래스 내 | 상속 관계 |

## 5. 객체지향 설계 5원칙 (SOLID) ★매우 중요★

| 원칙 | 약자 | 설명 |
| :--- | :--- | :--- |
| **단일 책임 원칙** | SRP | 객체는 단 **하나의 책임**만 가져야 한다 |
| **개방-폐쇄 원칙** | OCP | 기존 코드를 변경하지 않고 기능을 추가할 수 있어야 한다 (확장에 Open, 수정에 Close) |
| **리스코프 치환 원칙** | LSP | 자식 클래스는 부모 클래스의 역할을 **완벽히 대체**할 수 있어야 한다 |
| **인터페이스 분리 원칙** | ISP | 자신이 사용하지 않는 인터페이스와 의존 관계를 맺거나 영향을 받지 않아야 한다 |
| **의존 역전 원칙** | DIP | 변화하기 어려운 것(추상 클래스, 인터페이스)에 의존해야 한다 |

## 6. GoF 디자인 패턴 (Design Pattern) ★매우 중요★
소프트웨어 설계 시 자주 발생하는 문제들을 해결하기 위해 고안된 재사용 가능한 설계 모형입니다. (총 23개)
*(암기: 생구행)*

### 생성 패턴 (Creational) - 5개
객체 생성에 관련된 패턴입니다. *(암기: 빌·프·팩·앱·싱)*

| 패턴 | 설명 |
| :--- | :--- |
| **Builder** | 복잡한 인스턴스를 조립하여 만드는 구조. 생성과 표기를 분리 |
| **Prototype** | 원본 객체를 **복제(Clone)**하여 생성 |
| **Factory Method** | 객체 생성을 **서브 클래스에서 위임**하여 처리 (가상 생성자) |
| **Abstract Factory** | 구체적인 클래스에 의존하지 않고 인터페이스를 통해 연관/의존 객체 그룹을 생성 |
| **Singleton** | 어떤 클래스의 인스턴스가 **오직 하나**임을 보장 |

### 구조 패턴 (Structural) - 7개
클래스나 객체를 조합해 더 큰 구조를 만드는 패턴입니다. *(암기: 어·브·컴·데·퍼·플·프)*

| 패턴 | 설명 |
| :--- | :--- |
| **Adapter** | 호환성이 없는 기존 클래스의 인터페이스를 **변환**하여 재사용 |
| **Bridge** | 구현부에서 추상층을 **분리**하여 각자 독립적으로 확장 |
| **Composite** | 객체들의 관계를 **트리 구조**로 구성하여 부분-전체 계층을 표현 |
| **Decorator** | 객체의 결합을 통해 기능을 **동적으로** 유연하게 확장 |
| **Facade** | 복잡한 서브 클래스들을 피해 상위에 **단순한 인터페이스** 제공 |
| **Flyweight** | 인스턴스를 가능한 한 **공유**해서 사용하여 메모리 절약 |
| **Proxy** | 특정 객체로의 접근을 제어하기 위한 **대리자** 역할 |

### 행위 패턴 (Behavioral) - 11개
객체나 클래스 사이의 알고리즘이나 책임 분배에 관련된 패턴입니다.

| 패턴 | 설명 |
| :--- | :--- |
| **Observer** | 한 객체의 **상태 변화 시** 의존하는 다른 객체들에게 **자동 통보** |
| **State** | 객체의 **상태에 따라** 행위 내용을 변경 |
| **Strategy** | 동일한 계열의 알고리즘을 **캡슐화**하여 상호 교체 가능하게 함 |
| **Template Method** | 상위 클래스에서 **골격(뼈대)**을 정의하고, 하위 클래스에서 세부 처리를 구현 |
| **Command** | 요청(명령)을 **객체로 캡슐화**하여 매개변수화, 로깅, 취소 가능하게 함 |
| **Iterator** | 컬렉션의 내부 구조를 노출하지 않고 **순차적으로 접근** |
| **Mediator** | 객체 간의 상호작용을 **중재자 객체**에 캡슐화하여 결합도↓ |
| **Memento** | 객체의 이전 상태를 **저장(기록)**하여 나중에 복원할 수 있게 함 (Undo 기능) |
| **Chain of Responsibility** | 요청을 처리할 수 있는 객체가 **둘 이상**이면 한 객체에서 처리 못 하면 **다음 객체로 전달** |
| **Visitor** | 처리 기능을 분리하여 별도의 클래스를 만들고, 해당 클래스의 메서드가 각 클래스를 **방문**하며 처리 |
| **Interpreter** | 문법 규칙을 클래스화한 구조. 언어의 **문법을 해석**하는 패턴 |

## ★ 핵심 암기 포인트 ★

1. **결합도 순서:** 데이터 < 스탬프 < 제어 < 외부 < 공통 < 내용 (낮을수록 좋음)
2. **응집도 순서:** 우연적 < 논리적 < 시간적 < 절차적 < 통신적 < 순차적 < 기능적 (높을수록 좋음)
3. **Fan-In 높게, Fan-Out 낮게**
4. **SOLID 원칙:** SRP(단일책임), OCP(개방폐쇄), LSP(리스코프치환), ISP(인터페이스분리), DIP(의존역전)
5. **오버로딩 vs 오버라이딩:** 오버로딩=같은 클래스/매개변수 다름, 오버라이딩=상속관계/재정의
6. **디자인 패턴 분류:** 생성(5) + 구조(7) + 행위(11) = 23개
7. **코드 종류:** 순차, 블록, 10진, 그룹분류, 연상, 표의숫자, 합성
8. **공통 모듈 명세 원칙:** 정확성, 명확성, 완전성, 일관성, 추적성
