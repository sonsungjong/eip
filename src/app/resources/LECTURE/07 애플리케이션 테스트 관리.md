# [7과목] 애플리케이션 테스트 관리

---

## 1. 소프트웨어 테스트 기본 원칙 (7가지)

| 원칙 | 설명 |
| :--- | :--- |
| **완벽한 테스팅은 불가능** | 모든 경로와 입력 조합을 100% 테스트하는 것은 불가능 |
| **결함 집중 (Defect Clustering)** | 결함의 대부분은 소수의 특정 모듈에 집중 (파레토 법칙, 80대 20 법칙) |
| **살충제 패러독스 (Pesticide Paradox)** | 동일한 테스트 케이스 반복 시 더 이상 새로운 결함을 발견할 수 없음. 주기적 테스트 케이스 리뷰·개선 필요 |
| **오류-부재의 궤변** | 결함을 모두 제거해도 사용자 요구사항을 충족하지 못하면 품질이 높다고 볼 수 없음 |
| **테스팅은 정황(Context)에 의존** | 소프트웨어의 특성, 환경, 리스크에 따라 테스트 방법이 달라져야 함 |
| **조기 테스팅 (Early Testing)** | 개발 초기에 테스팅을 시작하여 결함을 조기에 발견해야 함 (결함 수정 비용↓) |
| **테스팅은 개발자와 별도의 팀에서 수행** | 개발자 본인이 아닌 독립적인 테스트 전문가가 수행하는 것이 효과적 |

---

## 2. 정적 테스트 vs 동적 테스트

| 구분 | 정적 테스트 | 동적 테스트 |
| :--- | :--- | :--- |
| **정의** | 프로그램을 **실행하지 않고** 소스 코드나 산출물을 검토 | 프로그램을 **실행하여** 결함을 발견 |
| **기법** | 리뷰, 인스펙션, 워크스루, 정적 분석 | 화이트박스 테스트, 블랙박스 테스트 |

### 정적 테스트 - 리뷰 기법

| 기법 | 설명 |
| :--- | :--- |
| **동료 검토 (Peer Review)** | 작성자가 동료에게 내용을 설명하고, 동료들이 결함을 발견하는 비공식적 검토 |
| **워크스루 (Walk Through)** | 작성자가 직접 산출물을 설명하면서 이해관계자들이 질문하고 결함을 찾는 검토 |
| **인스펙션 (Inspection)** | 전문 검토자(Moderator)가 주도하여 체크리스트 기반으로 **공식적**으로 결함을 검출하는 가장 체계적인 검토 방법 |

---

## 3. 화이트박스 테스트 (White Box Testing)
개발자 관점에서 소스 코드의 **내부 논리적인 구조와 동작**을 검사하는 구조 기반 테스트입니다.

### 테스트 커버리지 종류

| 커버리지 | 설명 |
| :--- | :--- |
| **구문(Statement) 커버리지** | 코드의 모든 **문장(구문)**이 최소 한 번 이상 실행되도록 설계 |
| **결정/분기(Decision/Branch) 커버리지** | 조건식의 전체 결과가 **참(T)과 거짓(F)**을 각각 최소 한 번 이상 갖도록 설계 |
| **조건(Condition) 커버리지** | 조건식 내부의 **개별 조건**들이 참과 거짓을 각각 최소 한 번 이상 갖도록 설계 |
| **조건/결정(Condition/Decision) 커버리지** | 개별 조건식과 전체 조건식이 **모두** 참/거짓을 한 번 이상 갖도록 설계 |
| **변경 조건/결정(MC/DC) 커버리지** | 각 개별 조건이 전체 조건식의 결과에 **독립적으로 영향**을 미치는지 확인 |
| **다중 조건(Multiple Condition) 커버리지** | 모든 개별 조건의 **가능한 조합**을 테스트. 가장 강력하지만 테스트 케이스 수가 기하급수적으로 증가 |

> **커버리지 강도:** 구문 < 결정 < 조건/결정 < MC/DC < 다중 조건

### 기본 경로 검사 (Base Path Testing) 
* 흐름도를 이용해 논리적 복잡성을 측정하여 독립적인 경로를 도출하고 테스트
* **McCabe의 순환 복잡도 공식:** **V(G) = E - N + 2**
  * E: 화살표(간선) 수, N: 노드 수
  * 순환 복잡도 = 독립적인 경로의 수 = 최소 테스트 케이스 수

## 4. 블랙박스 테스트 (Black Box Testing)
사용자 관점에서 **입력에 대한 올바른 출력이 나오는지 기능(명세서)을 검사**하는 테스트입니다.

### 블랙박스 테스트 종류
*(암기: 동·경·원·오·비·결)*

| 기법 | 설명 |
| :--- | :--- |
| **동치 분할 검사 (Equivalence Partitioning)** | 입력 데이터 영역을 **유효값/무효값**으로 나누어 대표값으로 테스트 |
| **경계값 분석 (Boundary Value Analysis)** | 입력 조건의 **경계값**에서 오류 발생 확률이 높음을 이용하여 테스트 |
| **원인-효과 그래프 (Cause-Effect Graphing)** | 입력(원인)과 출력(효과)의 관계를 논리적 그래프로 표현 |
| **오류 예측 검사 (Error Guessing)** | 테스터의 **경험과 직관**을 바탕으로 오류를 예측하여 테스트 |
| **비교 검사 (Comparison Testing)** | 여러 버전의 프로그램에 동일 데이터를 제공하여 결과를 비교 |
| **결정 테이블 검사 (Decision Table Testing)** | 입력 조건과 행위의 모든 조합을 **테이블**로 작성하여 테스트 |

## 5. 개발 단계에 따른 테스트 (V-모델)
*(순서 암기: 단 → 통 → 시 → 인)*

| 테스트 레벨 | 설명 | 대응 개발 단계 |
| :--- | :--- | :--- |
| **단위 테스트 (Unit Test)** | 모듈/컴포넌트 단위로 수행. 주로 화이트박스 테스트 | 코딩 |
| **통합 테스트 (Integration Test)** | 모듈 결합 시 인터페이스 연동 오류를 찾는 테스트 | 설계 |
| **시스템 테스트 (System Test)** | 전체 시스템이 요구사항에 맞게 수행되는지 점검 | 분석 |
| **인수 테스트 (Acceptance Test)** | 사용자 요구사항 충족 여부를 최종 확인 | 요구사항 |

### 인수 테스트 유형
| 유형 | 설명 |
| :--- | :--- |
| **알파 테스트** | **개발자 환경**에서 사용자가 개발자 앞에서(통제하에) 수행 |
| **베타 테스트** | **실제 사용자 환경**에서 다수의 사용자가 통제 없이 수행 |

---

## 6. 통합 테스트의 종류

| 방식 | 설명 | 가짜 모듈 |
| :--- | :--- | :--- |
| **하향식 (Top-Down)** | 상위 모듈 → 하위 모듈 방향으로 통합. 깊이 우선/넓이 우선 방식 | **스텁(Stub)** 사용 |
| **상향식 (Bottom-Up)** | 하위 모듈 → 상위 모듈 방향으로 통합. 클러스터 단위로 결합 | **드라이버(Driver)** 사용 |
| **샌드위치 (Sandwich)** | 하향식 + 상향식을 결합한 하이브리드 방식 | 스텁 + 드라이버 |
| **빅뱅 (Big Bang)** | 모든 모듈을 한꺼번에 통합하여 테스트 | 결함 위치 파악 어려움 |

> **암기: 하-스(하향식-스텁), 상-드(상향식-드라이버)**

## 7. 회귀 테스트 (Regression Test)
소프트웨어 변경(수정, 추가) 후 변경 부분이 **기존 기능에 영향을 미치지 않았는지** 확인하기 위해 기존 테스트 케이스를 재실행하는 테스트입니다.
* 살충제 패러독스와 연관: 회귀 테스트 시에도 테스트 케이스를 주기적으로 갱신해야 함

---

## 8. 테스트 하네스 (Test Harness)
테스트를 지원하기 위해 만든 코드와 데이터의 모음(도구)입니다.

| 구성요소 | 설명 |
| :--- | :--- |
| **테스트 드라이버 (Driver)** | 상위 모듈 역할을 하는 가짜 프로그램. 하위 모듈을 호출 |
| **테스트 스텁 (Stub)** | 하위 모듈 역할을 하는 가짜 프로그램. 상위 모듈로부터 호출됨 |
| **테스트 슈트 (Suite)** | 테스트 케이스의 집합 |
| **테스트 케이스 (Case)** | 테스트 항목의 명세서. 입력값, 실행 조건, 기대 결과 등을 포함 |
| **테스트 시나리오 (Scenario)** | 테스트 수행을 위한 여러 테스트 케이스의 **동작 순서**를 기술한 문서 |
| **테스트 스크립트 (Script)** | 테스트 케이스의 실행 절차를 자동화하기 위해 작성한 프로그램 |
| **목 오브젝트 (Mock Object)** | 사전에 사용자의 행위를 조건부로 입력해 두면 예정된 행위를 수행하는 객체 |

### 테스트 케이스 구성요소
| 구성요소 | 설명 |
| :--- | :--- |
| **식별자 (Identifier)** | 테스트 케이스를 구분하는 고유 번호 |
| **테스트 항목 (Test Item)** | 테스트 대상 (모듈, 기능 등) |
| **입력 명세 (Input Specification)** | 테스트에 사용할 입력 데이터 |
| **출력 명세 (Output Specification)** | 테스트 실행 후 기대되는 결과 |
| **환경 설정 (Environmental Needs)** | 테스트 수행에 필요한 하드웨어·소프트웨어 환경 |
| **특수 절차 요구 (Special Procedure Requirements)** | 테스트 케이스 수행 시 특별히 요구되는 절차 |
| **의존성 기술 (Inter-case Dependencies)** | 테스트 케이스 간의 의존성 |

## 9. 테스트 오라클 (Test Oracle)
테스트 결과가 참인지 거짓인지를 판단하기 위해 사전에 정의된 참값과 비교하는 기법입니다.

| 종류 | 설명 |
| :--- | :--- |
| **참(True) 오라클** | **모든** 입력값에 대해 기대 결과를 생성. 오류를 모두 검출 가능 |
| **샘플링(Sampling) 오라클** | **특정 몇 개**의 입력값에 대해서만 기대 결과를 제공 |
| **휴리스틱(Heuristic) 오라클** | 샘플링 오라클을 개선. 특정 입력은 올바른 결과, 나머지는 추정으로 처리 |
| **일관성 검사(Consistent) 오라클** | 변경 전후의 결괏값이 **동일한지** 확인 |

## 10. 결함 관련 용어

### 오류 → 결함 → 실패 관계
| 용어 | 설명 |
| :--- | :--- |
| **오류 (Error)** | 개발자의 실수로 인해 발생. 결함의 원인 |
| **결함 (Defect/Bug)** | 오류로 인해 소프트웨어 산출물에 포함된 문제. 실패의 원인 |
| **실패 (Failure)** | 결함이 실행되어 소프트웨어가 기대와 다르게 동작하는 것 |

> **흐름: 오류(Error) → 결함(Defect) → 실패(Failure)**

### 결함 심각도 (Severity)
| 등급 | 설명 |
| :--- | :--- |
| **치명적 (Critical)** | 시스템 전체가 작동 불능 |
| **주요 (Major)** | 주요 기능에 영향. 대체 경로 존재 가능 |
| **보통 (Normal)** | 일부 기능에 영향 |
| **경미 (Minor)** | 사용에 거의 영향 없음. UI 등 외관상 문제 |

## ★ 핵심 암기 포인트 ★

1. **테스트 원칙 7가지:** 완벽 불가능, 결함 집중(파레토), 살충제 패러독스, 오류-부재의 궤변, 정황 의존, 조기 테스팅, 독립적 수행
2. **블랙박스 기법:** 동치 분할, 경계값 분석, 원인-효과, 오류 예측, 비교 검사
3. **V-모델:** 단위 → 통합 → 시스템 → 인수 테스트
4. **하-스(하향식-스텁), 상-드(상향식-드라이버)**
5. **McCabe 순환 복잡도:** V(G) = E - N + 2
6. **오류 → 결함 → 실패** 순서
7. **테스트 하네스 구성:** 드라이버, 스텁, 슈트, 케이스, 시나리오, 스크립트, 목 오브젝트
8. **정적 테스트 리뷰:** 동료 검토(비공식), 워크스루(작성자 설명), 인스펙션(공식/체계적)
9. **커버리지 강도:** 구문 < 결정 < 조건/결정 < MC/DC < 다중 조건
10. **테스트 케이스 구성요소:** 식별자, 테스트 항목, 입력 명세, 출력 명세, 환경 설정, 특수 절차 요구, 의존성 기술
