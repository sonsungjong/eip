# [3과목] 통합 구현

---

## 1. 내외부 연계 메커니즘
서로 다른 두 시스템이나 소프트웨어를 연결하여 데이터를 교환하고 연동하는 방식입니다.

### 직접 연계 방식
중간 매개체 없이 직접 연계하는 방식.

| 기술 | 설명 |
| :--- | :--- |
| **DB Link** | 데이터베이스에서 제공하는 DB Link 객체를 이용하여 수신 시스템의 DB에 직접 접근하는 방식 |
| **DB Connection** | 수신 시스템의 WAS에서 송신 시스템의 DB로 직접 연결하는 방식 (DBMS Connection Pool 이용) |
| **API / Open API** | 송신 시스템의 데이터베이스에서 데이터를 읽어와 제공하는 인터페이스 프로그램 |
| **JDBC** | Java 언어로 다양한 종류의 데이터베이스에 접속하고 SQL문을 수행할 때 사용되는 표준 API |
| **Hyper Link** | 현재 텍스트에서 다른 텍스트(문서)로 이동하는 연결 기능 |

* 장점: 개발비용 저렴, 연계 처리 속도 빠름.
* 단점: 시스템 간 결합도가 높아 변경에 민감, 보안에 취약함.

### 간접 연계 방식
중간에 연계 서버(미들웨어)를 두어 연계하는 방식. (예: EAI, ESB, Socket)

* 장점: 서로 다른 네트워크/프로토콜 연계 가능, 인터페이스 변경에 유연함.
* 단점: 연계 구조 복잡, 성능 저하 우려, 테스트 기간이 김.

## 2. 미들웨어 (Middleware) 유형
운영체제와 응용 프로그램 사이에서 추가적인 서비스를 제공하는 소프트웨어입니다.

| 유형 | 설명 |
| :--- | :--- |
| **DB** | 데이터베이스 벤더에서 제공하는 클라이언트-서버 간 미들웨어 (예: ODBC, JDBC) |
| **RPC (Remote Procedure Call)** | 원격 프로시저를 로컬 프로시저처럼 호출하는 미들웨어 |
| **MOM (Message Oriented Middleware)** | 메시지 기반의 **비동기형** 미들웨어. 즉각적인 응답을 필요로 하지 않는 환경에 적합 |
| **TP Monitor (Transaction Processing Monitor)** | 온라인 트랜잭션을 처리 및 감시하는 미들웨어. 항공기 예약, 은행 등 대규모 거래 처리에 사용 |
| **ORB (Object Request Broker)** | 객체 지향 미들웨어. CORBA 표준 스펙을 구현한 미들웨어 |
| **WAS (Web Application Server)** | 사용자의 요구에 따라 변하는 동적 콘텐츠를 처리하기 위한 미들웨어 (예: Tomcat, WebLogic, JBoss) |

---

## 3. EAI (Enterprise Application Integration)
기업 내의 서로 다른 애플리케이션 및 시스템 간에 데이터를 원활하게 통신할 수 있도록 묶어주는 통합 솔루션입니다.

### EAI 구축 유형
| 유형 | 설명 | 특징 |
| :--- | :--- | :--- |
| **Point-to-Point** | 1:1 단순 통합 방식 | 가장 기초적인 애플리케이션 통합 방법. 변경·재사용 어려움 |
| **Hub & Spoke** | 단일 접점인 허브 시스템을 통해 데이터를 전송하는 중앙 집중식 방식 | 허브 장애 시 전체 시스템에 영향을 미침 |
| **Message Bus** | 애플리케이션 사이에 미들웨어(버스)를 두어 연계하는 방식 | 뛰어난 확장성, 대용량 데이터 처리에 용이 |
| **Hybrid** | 그룹 내는 Hub & Spoke, 그룹 간에는 Message Bus 방식을 사용하는 통합 방식 | 병목 현상 최소화 |

---

## 4. ESB (Enterprise Service Bus)
애플리케이션 간 연계, 데이터 변환, 웹 서비스 지원 등 표준 기반의 인터페이스를 제공하는 솔루션입니다.

* EAI와 비슷하지만, 애플리케이션보다는 **서비스(Service)** 중심으로 통합을 지향하는 아키텍처(SOA)입니다.
* 특정 서비스에 국한되지 않고 범용적으로 사용 가능하며, 결합도를 낮게(Loosely Coupled) 유지합니다.

### EAI vs ESB 비교
| 구분 | EAI | ESB |
| :--- | :--- | :--- |
| 통합 방식 | 애플리케이션 중심 (Point-to-Point) | 서비스 중심 (Bus 방식) |
| 결합도 | 높음 (Tightly Coupled) | 낮음 (Loosely Coupled) |
| 표준 | 벤더 종속적 | 표준 기술 기반 (XML, 웹 서비스) |
| 확장성 | 제한적 | 유연한 확장 가능 |

---

## 5. 웹 서비스 (Web Service) ★중요★
네트워크 상에서 서로 다른 종류의 컴퓨터들 간에 상호작용을 하기 위한 소프트웨어 시스템입니다.

* **SOAP (Simple Object Access Protocol):** HTTP, HTTPS, SMTP 등을 사용하여 **XML** 기반의 메시지를 네트워크 상태에서 교환하는 **프로토콜**.
* **WSDL (Web Services Description Language):** 웹 서비스에 대한 상세 정보(서비스명, 제공 위치, 메시지 포맷, 프로토콜 등)를 기술한 XML 형식으로 구현된 **언어**.
* **UDDI (Universal Description, Discovery and Integration):** WSDL을 등록하고 검색하기 위한 저장소로, 공개적으로 접근 및 검색이 가능한 **레지스트리**.

> **웹 서비스 동작 흐름:**
> 서비스 제공자가 UDDI에 WSDL을 등록 → 서비스 요청자가 UDDI에서 WSDL을 검색 → SOAP 프로토콜을 이용하여 서비스 제공자와 통신

---

## 6. REST (Representational State Transfer)
웹과 같은 분산 하이퍼미디어 환경에서 자원의 존재/상태를 전달하기 위한 네트워크 아키텍처입니다.

* 자원(Resource)을 **URI**로 정의하고, 자원에 대한 상태(행위)를 **HTTP Method**로 명시합니다.

### HTTP Method와 CRUD 매핑 ★빈출★
| HTTP Method | CRUD | 설명 |
| :--- | :--- | :--- |
| **POST** | Create | 자원 생성 |
| **GET** | Read | 자원 조회 |
| **PUT** | Update | 자원 수정 |
| **DELETE** | Delete | 자원 삭제 |

### REST 특징
* **무상태성(Stateless):** 서버가 클라이언트의 상태를 보존하지 않음
* **캐시 처리 가능(Cacheable):** HTTP가 가진 캐싱 기능 적용 가능
* **클라이언트-서버 구조:** 자원을 가진 서버와 자원을 요청하는 클라이언트로 구분
* **균일한 인터페이스(Uniform Interface):** URI로 지정한 자원에 대한 조작을 통일된 인터페이스로 수행

## 7. 주요 데이터 교환 형식

| 구분 | JSON | XML |
| :--- | :--- | :--- |
| **정식 명칭** | JavaScript Object Notation | eXtensible Markup Language |
| **형태** | 속성-값 쌍 (Key-Value Pair) | 사용자 정의 태그 기반 마크업 |
| **특징** | 가벼움, AJAX에서 주로 사용 | 확장성, 구조적 데이터 전달에 적합 |
| **가독성** | 간결하여 가독성이 좋음 | 태그가 많아 상대적으로 복잡 |

## 8. 연계 테스트 ★서술형 대비★
연계 모듈 구현 후 데이터가 정상적으로 전달되는지 확인하는 과정입니다.

### 연계 테스트 절차
1. **연계 테스트 케이스 작성:** 연계 시나리오에 기반하여 입력 데이터, 예상 결과, 테스트 조건 등을 정의
2. **연계 테스트 수행:** 작성된 테스트 케이스를 기반으로 연계 테스트를 실행
3. **연계 테스트 수행 결과 검증:** 실제 결과와 예상 결과를 비교하여 정상 동작 여부를 확인

---

## 9. 인터페이스 보안

### 데이터 전송 구간 보안
* **네트워크 영역:** IPSec, SSL/TLS 등 보안 프로토콜을 적용하여 데이터를 암호화 전송
* **애플리케이션 영역:** 시큐어 코딩(Secure Coding) 가이드를 적용하여 보안 취약점 제거

### 데이터베이스 암호화 방식
| 방식 | 설명 |
| :--- | :--- |
| **API 방식** | 애플리케이션 레벨에서 암호 모듈(API)을 적용하는 방식 |
| **Plug-in 방식** | DB 서버에 암·복호화 모듈을 설치하는 방식 |
| **Hybrid 방식** | API + Plug-in 방식을 결합한 방식 |

### 인터페이스 감시 도구
* **스카우터(Scouter):** 애플리케이션 및 시스템 모니터링 도구
* **제니퍼(Jennifer):** 실시간 트랜잭션 모니터링 및 분석 도구

## 10. IPC (Inter-Process Communication)
프로세스 간 통신에서 프로세스들 사이에 데이터를 주고받는 행위 또는 그에 대한 방법입니다.

| 방식 | 설명 |
| :--- | :--- |
| **공유 메모리 (Shared Memory)** | 여러 프로세스가 동일한 메모리 영역을 공유하여 데이터를 교환하는 방식 |
| **소켓 (Socket)** | 네트워크를 통해 서로 다른 컴퓨터(또는 프로세스) 간 통신을 지원하는 방식 |
| **세마포어 (Semaphore)** | 공유 자원에 대한 접근을 제어하기 위해 사용되는 동기화 기법 (상호배제) |
| **파이프 (Pipe)** | 하나의 프로세스가 쓰고 다른 프로세스가 읽는 단방향 통신 방식 |
| **메시지 큐 (Message Queue)** | 메시지를 큐에 저장하고 순서대로 읽는 비동기 통신 방식 |

## ★ 핵심 암기 포인트 ★

1. **EAI 4가지 유형:** Point-to-Point → Hub & Spoke → Message Bus → Hybrid
2. **웹 서비스 3요소:** SOAP(프로토콜), WSDL(언어), UDDI(레지스트리)
3. **REST HTTP Method-CRUD 매핑:** POST=Create, GET=Read, PUT=Update, DELETE=Delete
4. **미들웨어 유형:** DB, RPC, MOM, TP Monitor, ORB, WAS
5. **직접 연계 vs 간접 연계:** 직접(빠르지만 결합도↑) / 간접(유연하지만 복잡)
6. **EAI vs ESB:** EAI는 애플리케이션 중심·높은 결합도, ESB는 서비스 중심·낮은 결합도
7. **연계 테스트 절차:** 케이스 작성 → 테스트 수행 → 결과 검증
