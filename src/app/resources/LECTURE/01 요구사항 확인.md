# [1과목] 요구사항 확인

## 1. 소프트웨어 생명 주기 (SDLC; Software Development Life Cycle)
소프트웨어를 개발하기 위한 설계, 운용, 유지보수 등의 전 과정을 각 단계별로 나눈 모델입니다.

* **폭포수 모형 (Waterfall):** 가장 오래된 전통적 방식. 이전 단계가 완벽히 끝나야 다음 단계로 넘어감 (선형 순차적). 요구사항 변경이 어려움.
    * 단계: `타당성 검토` ➔ `계획` ➔ `요구사항 분석` ➔ `설계` ➔ `구현` ➔ `테스트` ➔ `유지보수`
* **프로토타이핑 모형 (Prototyping):** 사용자의 요구사항을 파악하기 위해 실제 개발될 소프트웨어에 대한 시제품(프로토타입)을 만들어 최종 결과물을 예측하는 모형.
* **나선형 모형 (Spiral):** 보헴(Boehm)이 제안. 반복적인 작업을 수행하며 점진적으로 개발. **위험 최소화**가 목적.
    * 절차: `계획 수립` ➔ `위험 분석` ➔ `개발 및 검증` ➔ `고객 평가`
* **애자일 모형 (Agile):** 고객의 요구사항 변화에 유연하게 대응할 수 있도록 일정한 주기를 반복하면서 개발하는 방식.
    * 종류: XP, 스크럼(Scrum), 칸반(Kanban), 크리스탈(Crystal), FDD 등

---

## 2. 애자일 (Agile) 방법론
문서화보다는 실질적인 코딩과 고객과의 소통을 중시하는 방법론입니다.

### XP (eXtreme Programming)
수시로 발생하는 고객의 요구사항에 유연하게 대응하기 위해 고객의 참여와 개발 과정의 반복을 극대화한 방법론.

* **XP의 5가지 핵심 가치:** 의사소통(Communication), 단순성(Simplicity), 용기(Courage), 존중(Respect), 피드백(Feedback)

* **XP의 주요 실천 방법 (Practice):**

| 실천 방법 | 설명 |
| :--- | :--- |
| **테스트 주도 개발 (TDD)** | 코드 작성 전 테스트 케이스를 먼저 작성 |
| **페어 프로그래밍** | 두 명의 개발자가 한 컴퓨터로 함께 개발 |
| **지속적 통합 (CI)** | 매일 여러 번 소스를 통합하고 빌드 |
| **리팩토링** | 기능 변경 없이 내부 구조를 개선 |
| **소규모 릴리즈** | 작은 단위로 자주 제품을 릴리즈 |
| **코드 공동 소유** | 모든 팀원이 코드 수정 가능 |
| **온사이트 고객** | 고객이 개발팀과 함께 상주 |
| **코딩 표준** | 팀 전체가 공통 코딩 규칙 준수 |
| **단순한 설계** | 현재 요구사항에만 집중한 최소 설계 |
| **주 40시간 작업** | 과도한 야근 금지, 지속 가능한 속도 유지 |

---

### 스크럼 (Scrum)
팀원 스스로가 스크럼 팀을 구성하고 개발 작업에 관한 모든 것을 스스로 해결할 수 있어야 하는 방법론.

* **주요 용어:**
    * **제품 백로그 (Product Backlog):** 개발에 필요한 모든 요구사항을 우선순위에 따라 나열한 목록.
    * **스프린트 (Sprint):** 2~4주 정도의 짧은 개발 기간.
    * **스프린트 백로그:** 스프린트 목표 달성을 위해 선택한 제품 백로그 항목.
    * **스크럼 미팅 (Daily Scrum):** 매일 15분 정도 진행하는 짧은 진행 상황 점검 회의.

* **스크럼 이벤트 4종:**

| 이벤트 | 설명 |
| :--- | :--- |
| **스프린트 계획 (Sprint Planning)** | 스프린트 목표 및 백로그 선정 |
| **일일 스크럼 (Daily Scrum)** | 매일 15분, 진행 상황 점검 |
| **스프린트 검토 (Sprint Review)** | 스프린트 결과물을 고객에게 시연 |
| **스프린트 회고 (Sprint Retrospective)** | 프로세스 개선을 위한 팀 내부 회의 |

## 3. 요구사항 (Requirements)
소프트웨어가 어떤 문제를 해결하기 위해 제공하는 서비스에 대한 설명과 정상적으로 운영되는데 필요한 제약조건입니다.

### 요구사항의 분류
| 구분 | 설명 | 예시 |
| :--- | :--- | :--- |
| **기능적 요구사항** | 시스템이 **무엇을 하는지**, 어떤 기능을 하는지에 대한 사항 | 결제 기능, 장바구니 추가, 회원가입 |
| **비기능적 요구사항** | 시스템의 장비, 성능, 인터페이스, 테스트, 보안 등 **품질**에 대한 제약사항 | 응답 시간은 2초 이내, 동시 접속자 1000명 수용 |

---

### 요구사항 개발 프로세스 (도-분-명-확)
1. **요구사항 도출 (Elicitation):** 요구사항이 어디에 있는지, 어떻게 수집할 것인지 식별 (인터뷰, 설문조사, 브레인스토밍).
2. **요구사항 분석 (Analysis):** 도출된 요구사항의 타당성을 조사하고 비용과 일정을 파악 (DFD, 자료 사전 등 사용).
3. **요구사항 명세 (Specification):** 분석된 요구사항을 바탕으로 모델 작성 및 문서화.
    * **정형 명세:** 수학적 기호·표기법을 사용 (Z 언어, VDM 등). 정확하지만 작성 어려움.
    * **비정형 명세:** 자연어, 그림 등을 사용. 이해하기 쉽지만 모호성 존재.
4. **요구사항 확인 (Validation):** 개발된 요구사항 명세서가 고객의 요구를 정확히 반영했는지 검토.

### 요구사항 확인 기법 3종 ★
| 기법 | 설명 |
| :--- | :--- |
| **동료 검토 (Peer Review)** | 요구사항 명세서 작성자가 직접 설명하고 동료들이 검토 |
| **워크스루 (Walk Through)** | 검토 자료를 사전 배포 후 회의를 통해 오류를 조기에 검출. 비공식적 검토 |
| **인스펙션 (Inspection)** | 작성자 외 전문 검토 그룹이 공식 절차에 따라 검토. 가장 공식적인 방법 |

## 4. UML (Unified Modeling Language)
시스템 개발 과정에서 의사소통이 원활하게 이루어지도록 표준화한 범용 모델링 언어입니다.

### UML의 구성 요소
* **사물 (Things):** 모델을 구성하는 가장 중요한 기본 요소.
* **관계 (Relationships):** 사물과 사물 사이의 연관성을 표현.
* **다이어그램 (Diagram):** 사물과 관계를 도형으로 표현한 것.

---

### UML 관계 6종 ★★★ (실기 빈출)
| 관계 | 설명 | 표기 |
| :--- | :--- | :--- |
| **연관 (Association)** | 클래스 간 서로 참조하는 관계 | 실선 |
| **집합 (Aggregation)** | 전체-부분 관계. 부분이 독립적으로 존재 가능 (약한 결합) | 빈 마름모 + 실선 |
| **포함 (Composition)** | 전체-부분 관계. 부분이 독립적으로 존재 불가 (강한 결합) | 채운 마름모 + 실선 |
| **의존 (Dependency)** | 한 클래스가 다른 클래스를 일시적으로 사용 | 점선 화살표 |
| **실체화 (Realization)** | 인터페이스를 구현(implement)하는 관계 | 점선 + 빈 삼각형 화살표 |
| **일반화 (Generalization)** | 상속(is-a) 관계. 부모↔자식 클래스 | 실선 + 빈 삼각형 화살표 |

> 💡 집합 vs 포함 구별: 자동차(전체)와 타이어(부분) → **집합** (타이어는 따로 존재 가능)
> 자동차(전체)와 엔진(부분) → **포함** (엔진은 자동차 없이 의미 없음)

---

### 유스케이스 다이어그램 관계 — 스테레오타입 ★★
| 스테레오타입 | 설명 |
| :--- | :--- |
| **«include»** | 반드시 포함되어 실행되는 관계 (필수적 포함) |
| **«extend»** | 특정 조건에서만 선택적으로 실행되는 관계 (선택적 확장) |

---

### 다이어그램의 종류
| 종류 | 다이어그램 명칭 | 설명 |
| :--- | :--- | :--- |
| **구조적(정적)** | **클래스 (Class)** | 시스템을 구성하는 클래스와 그들 간의 관계 표현 |
| | **객체 (Object)** | 인스턴스를 특정 시점의 객체와 객체 사이의 관계로 표현 |
| | **컴포넌트 (Component)** | 실제 구현 모듈인 컴포넌트 간의 관계 표현 |
| | **배치 (Deployment)** | 결과물, 프로세스, 컴포넌트 등 물리적 요소들의 위치 표현 |
| **행위적(동적)** | **유스케이스 (Use Case)** | 사용자 관점에서 시스템의 활동을 표현 |
| | **시퀀스 (Sequence)** | 객체 간에 주고받는 메시지의 순서를 시간의 흐름에 따라 표현 |
| | **상태 (State)** | 객체가 속해 있는 클래스의 상태 변화를 표현 |
| | **활동 (Activity)** | 시스템이 어떤 기능을 수행하는지 처리 로직이나 흐름을 표현 |

## 5. UI (User Interface) 설계
사용자와 시스템 간의 의사소통을 돕는 물리적, 가상적 매개체입니다.

### UI 유형
| 유형 | 설명 | 예시 |
| :--- | :--- | :--- |
| **CLI (Command Line Interface)** | 텍스트 명령어 입력 방식 | 터미널, CMD |
| **GUI (Graphical User Interface)** | 그래픽 기반 아이콘 조작 방식 | Windows, macOS |
| **NUI (Natural User Interface)** | 신체 동작·음성 인식 방식 | 터치스크린, 음성인식 |
| **VUI (Voice User Interface)** | 음성 명령 방식 | Siri, Alexa |

### UX vs UI 차이
* **UI (User Interface):** 사용자가 시스템과 상호작용하는 **화면·요소** 자체.
* **UX (User Experience):** 사용자가 시스템을 사용하면서 느끼는 **총체적 경험**. UI를 포함하는 더 넓은 개념.

### UI 설계 도구
| 도구 | 설명 |
| :--- | :--- |
| **와이어프레임 (Wireframe)** | 기획 단계에서 레이아웃을 선(wire)으로만 표현한 뼈대 |
| **목업 (Mockup)** | 실제 화면처럼 시각적으로 구현한 정적인 모형 |
| **프로토타입 (Prototype)** | 실제 동작하는 것처럼 인터랙션을 구현한 시제품 |
| **스토리보드 (Storyboard)** | 화면 흐름과 UI 요소를 순서대로 정리한 문서 |

### UI 설계 원칙 (직.유.학.유)
* **직관성 (Intuitiveness):** 누구나 쉽게 이해하고 사용할 수 있어야 함.
* **유효성 (Validity):** 사용자의 목적을 정확하게 달성할 수 있도록 유용해야 함.
* **학습성 (Learnability):** 누구나 쉽게 배우고 익힐 수 있어야 함.
* **유연성 (Flexibility):** 사용자의 요구사항을 최대한 수용하며, 오류를 최소화해야 함.
